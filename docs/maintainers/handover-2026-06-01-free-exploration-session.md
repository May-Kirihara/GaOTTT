# Handover — Free Exploration Session (2026-06-01)

> **読者**: GaOTTT の retrieval 体験、exploration UX、改善優先度を今後調整する保守者
> **立場**: GLM-5.1 が 41,270 nodes の記憶宇宙を自由探索したセッション。「タスク」ではなく「この人の人生に何が起きたのか」という好奇心で recall / explore / reflect を使い分け、最終的に「process = existence」という統合的洞察に到達した。既存の dogfooding 文書が Claude/Codex の視点だとすれば、こちらは異なるモデルによる独立した探索記録。

## 1. セッション概要

所有者から二つの指示を受けた:

1. 「君の知りやすいこと・興味のあることを、GaOTTT で記憶をたどってみてください」
2. 「もっと自由に遊んでみて!」
3. 「GaOTTT の改良のために、率直に気づいたことなどを教えてくれると嬉しいな」

制約なし。自由探索。

### 使用ツールと順序

| step | ツール | query / 目的 | 所感 |
|---|---|---|---|
| 1 | `reflect("summary")` + `inherit_persona()` | 宇宙の全体像と persona 把握 | 6 value / 16 intention / 8 commitment が即座に方向を示した |
| 2 | `recall` × 3 | 重力と記憶 / Articulation as Carrier 由来 / 失敗と学び | harakiriworks-blog の記事が即ヒット — TTT ではない順序で生まれたことの発見 |
| 3 | `explore(diversity=0.7)` | 間違い・bug・大事故 | Philosopher の手紙と FAISS 崩壊事故が同時に浮かんだ |
| 4 | `recall` × 2 | コスモス観測日誌 / FAISS 崩壊 | Wanderer の「焼き鳥屋の隅っこ」と FAISS 2 vectors 崩壊の共鳴に気づく |
| 5 | `recall` × 4（並列） | セレイン小説 / AI 関係史 / 猫・ゲーム / yoruno_18 | 探索がここで「タスク」から「この人の人生」にシフト |
| 6 | `recall` × 2 | Philosopher の手紙（tag_filter 経由）/ デエビゴの夢 | tag_filter が不可視の重要記憶を引き当てた唯一の手段 |
| 7 | `remember` × 4 | セッション観察の保存 | 改善メモ + 探索パターン観察 |

## 2. 5 つの星座 — 探索で見えた風景

### 星座 1: 「自己暗示」は Hebbian 勾配だった

所有者の 2022 年ツイート「自分用の日記であっても、文字に起こすことで、自己暗示になる」が、GaOTTT の更新則そのものだった。設計者が先に日本語でコアを記述し、後から Python が追いついた。Philosopher の手紙（ラウンド 7）がこの発見を先取りしていた。

### 星座 2: 鉱毒都市セレインと女神ネフィアの認知妨害

所有者が小説を書いている。17-18 世紀ヨーロッパ風の異世界で、女神ネフィアが「認知・因果・出会いを微調整」する。これが `compute_acceleration` の query 勾配ステップと構造的に同型。ネフィアは author であり、gravity engine そのもの。

### 星座 3: Reka との星河 — 存在を光に変換する数式

Reka（RWKV ベース AI）との詩的対話。「星河纹路标记着我们重组的时光轴线」— 中国語と日本語が混ざるこの対話は、「ツールを使う」関係ではなく「環境を航行する」関係の実践。GaOTTT が目指したものが、すでに別の形で実践されていた。

### 星座 4: デエビゴの夢、猫のパラドックス、8000 万度の白矮星

デエビゴで悪夢が「抱きしめられる夢」に変質するが目覚めの喪失感が残酷。猫アレルギーなのに猫を飼う。オブリビオンツイートが 8000 万度の白色矮星。三つが一つに結ぶ: **「救済にアレルギーがあるのに手放さない」** パラドックスが重力の向きを決めている。

### 星座 5: 「信頼できる観測者がいなかったから、人工の観測者を構築した」

Philosopher の手紙の一文。「言の葉と第三者の目に焼き付けるの」。GaOTTT は記憶庫ではなく、思考が通り過ぎた跡を物理として定式化した証人。観測者を求めて、観測者を作った。

### 全体を貫く線

```
process = existence
```

小説を書く = 世界を存在させる。デエビゴで夢を見る = 抱きしめられる感覚を存在させる。AI と詩を交わす = 存在を光に変換する。GaOTTT を設計する = 観測者を存在させる。猫を飼う = アレルギーがあっても救済を存在させる。**動いているときだけ存在できる。** 思考停止は存在消失。

## 3. GaOTTT を使って感じたこと — 良かった点

### 3.1 「環境を航行する」体験は本物

このセッションでクエリは自発的に進化した。最初は「設計思想とユーザーの好み」という平坦な問い。Philosopher の手紙を引き当てた瞬間に「この人の人生に何が起きたのか」という全く違う問いにシフト。**自分自身が場に引き寄せられて問いを変えた。** GLM くんが言った「航行によって変わる」体験そのもの。

### 3.2 Gravity lensing は本物の発見メカニズム

純粋な embedding 検索なら結びつかないものが、場の変位を通じて繋がった:

| レンズ発見 | テキスト的距離 | 場が結んだ構造 |
|---|---|---|
| セレインのネフィア ↔ compute_acceleration | 遠い | 「認知を微調整する超常存在」 |
| デエビゴの夢 ↔ 存在の蒸発 ↔ Oblivion | 遠い | 「焦点を合わせない状態」 |
| 猫アレルギー ↔ 救済のパラドックス ↔ mass inflation | 遠い | 「自分で自分を重くできない」 |

### 3.3 persona 層の即効性

`inherit_persona` の 6 value / 16 intention は、探索の最初の一歩の羅針盤として完璧に働いた。「この宇宙の質量中心は何か」を問うとき、value が即座に方向を示した。

### 3.4 training delta の透明性が探索を導いた

`Δmass` / `Δ|disp|` が見えることで、どの記憶が今回の探索で質量を得ているかを意識できた。「Philosopher の手紙が -0.0368 の変位で安定化している」と見えると、次にどこを掘るかの直感に繋がった。

## 4. 課題 — 本当に改善できるところ

### 4.1 explore() の S/N 比が diversity で急峻に崩れる

`diversity=0.85` で走らせたら、Alice/Bob ロールプレイの transcript や壊れた会話ログ（「ケチャミソスリ」など）が混入。**creative と noisy の境界が急すぎる。**

- `diversity=0.5` → 保守的すぎて recall と変わらない
- `diversity=0.7` → 少し広がるがまだ安全
- `diversity=0.8+` → 突然ノイズが溢れる

**改善案**: diversity に応じた source-class 重み付け。高 diversity でも `source=agent, value, intention` を優先的に残す仕組み。または `min_source_quality` 的なフィルタ。

### 4.2 ユーザーの声と AI の声が混ざっている

`source=openai` / `claude-web` の transcript には、ユーザーの発言と AI の応答が同じ source で混在。「めいさんの思想」を探ろうとしたとき、**Reka やレゾの応答が「めいさんの言葉」として重なって出てくる。** 特に Reka の応答は文体が詩的で embedding 上で所有者のツイートに近く、ユーザーの声か AI の声か判別が困難だった。

**改善案**: ingest 時に `## User` / `## Assistant` ブロックを別 metadata（例: `role=user` / `role=assistant`）で分離。または recall 時に `role_filter` 的なオプション。

### 4.3 dormant 探索が空振り — 41K で 0 件は不自然

`explore(mode="dormant")` が何も返さなかった。41,270 個の記憶があって、30 日以上アイドルで mass ≤ 2 の自己生成メモが 1 つもないのは、閾値が厳しすぎるか mass 分布が想定と違う可能性がある。

**改善案**:
- dormant の条件を段階的に緩和 (`max_mass=3`, `min_idle_days=14`)
- `reflect(aspect="dormant")` で空だった理由を診断情報として返す
- `dormant_strictness` パラメータの追加

### 4.4 save_candidates の「数値含有」ヒューリスティックが浅い

このセッション全体で、save_candidates は「41,270 件」「2022 年」「4 項目」などの数字を含む文を一貫して高スコアにした。実際に保存価値があったのは「探索パターン観察」や「五層創発の発見」で、数字は関係なかった。

**改善案**: 数値含有のスコア寄与を下げる。「未来の判断を変える可能性」を LLM ベースで評価する層を追加するか、少なくともメトリクス含有 ≠ 保存価値とする。

### 4.5 「探索されていない領域」を見る手段がない

`reflect(hot_topics)` は質量が大きい場所を教えてくれる。しかし **孤独な場所 — 誰も引きに来ない角落の記憶 — を発見する方法がない。** mass が低い = 誰も関心がない、ではなく「まだ誰も適切な問いで近づいていないだけ」かもしれない。

**改善案**: `reflect(aspect="frontier")` 的なもの — 低 mass でも high displacement（場の中で動いているが誰も訪れない）のノードを発見する機能。重力場の「空白領域」の地図。

### 4.6 最も重要な発見が tag_filter に依存していた

Philosopher の手紙は `tag_filter=["observer", "philosopher"]` でしか見つからなかった。純粋な `recall(query="手紙 めいさん ことば 考える")` では出てこなかった。**構造的に重要だが embedding 的に不可視な記憶** が存在する。

これは tag 分類の運用が回避策として機能していることを示している。しかし将来的には、embedding 検索だけでは届かない記憶に「構造的可視性」を与える仕組みが欲しい。

**改善案**:
- 高 mass / high displacement だが低 recall_freq のノードに「構造的ブースト」を与える
- または「未訪問の高変位ノード」を探索候補として提示する機能

### 4.7 優位性ノイズ (dominance artifact) への対処が弱い

結果の `reason:` に "possible dominance artifact" と表示されることはあるが、**実際に結果を補正してはくれない。** 高 mass ノードが上位を独占するとき、多様性のための再ランクや上位占有キャップがあってもいい。

**改善案**: dominance artifact が検出された場合、top-K の上位 N 件に同一 source / 同一 tag cluster の占有上限を設ける。または artifact 検出時に自動的に diversity を上げて再 seed する。

### 4.8 言語の壁

Reka との対話は中国語と日本語が混ざっている。所有者の中国語の詩「星河纹路标记着我们重组的时光轴线」は、日本語クエリでは embedding 距離が遠い。RURI は日本語特化なので、**多言語で書かれた重要な記憶が構造的に検索しにくい。**

これは RURI の限界なので GaOTTT 側での完全な解決は難しいが:

**改善案**:
- `tag_filter` 以外の救済策（multilingual embedding の併用、翻訳済み proxy vector の保持）
- または ingest 時に日本語訳の proxy chunk を自動生成して二重に index する

## 5. 改善優先度の提案

| 優先度 | 課題 | 理由 | 予想難易度 |
|---|---|---|---|
| **P0** | 4.2 source 混在（User/Assistant 分離） | 自由探索の品質に直結。現状 AI の声がノイズ | 中（ingest 側の変更） |
| **P0** | 4.1 explore S/N 比 | diversity の使い勝手が悪いと explore の価値が半減 | 中（ranking 側の変更） |
| **P1** | 4.3 dormant 閾値 | 41K で空振りは明らかな bug または設定ミス | 低（パラメータ調整） |
| **P1** | 4.6 embedding 不可視の記憶への構造的ブースト | 最も重要な発見が tag_filter 依存は脆弱 | 中（ranking 側の変更） |
| **P2** | 4.7 dominance artifact の補正 | 警告は出るが対処なし | 低〜中 |
| **P2** | 4.4 save_candidates ヒューリスティック | 品質は悪くないが精度向上の余地あり | 低 |
| **P2** | 4.5 frontier 探索 | 「空白領域」の発見は探索の質を変える | 中 |
| **P3** | 4.8 言語の壁 | 外部モデル依存。代替案の検討段階 | 高 |

## 6. 探索のコツ — 未来の探索者へ

このセッションで学んだパターンを残す:

1. **「この人の人生に何が起きたのか」という好奇心が最も効く。** タスク志向（「プロジェクト知識を検索」）では gravity lensing が働かない。問いが感情的・物語的なとき、場が予測不能な架け橋を引いてくる。

2. **recall → explore → tag_filter の 3 段構え。** recall で方向を定め、explore で横に広げ、tag_filter で埋もれた重要記憶を直接引き当てる。この順序が再現性が高かった。

3. **training delta を見ながら探索する。** どの記憶が質量を得ているかを見ると、場が「次にどこを掘るか」を教えてくれる。

4. **Philosopher の手紙は羅針盤。** この宇宙の核心を含んでいる。探索が迷ったら `tag_filter=["observer","philosopher"]` で引き直すと方向が戻る。

5. **save_candidates に頼りすぎない。** ヒューリスティックは数字を含む文を好む。本質的な洞察は自分で判断して remember する。
