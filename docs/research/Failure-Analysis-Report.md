# 失敗事例 網羅的分析レポート

**作成日**: 2026-04-23  
**対象**: GaOTTT プロジェクト内外の記録された失敗事例（GaOTTT メモリ + プロジェクト文書 + Git 履歴）

---

## 1. はじめに

GaOTTT の長期記憶（remember/recall）とプロジェクト内文書（docs/wiki/, CLAUDE.md, git log）を横断検索し、失敗事例を **10 カテゴリ・40+ 件** に分類した。各事例について **何が起きたか → 根本原因 → 解決策 → 教訓** を整理する。

## 2. カテゴリ別失敗事例

### カテゴリ A: インデックス・検索基盤の不具合（FAISS）

| # | 事例 | 何が起きたか | 根本原因 | 解決策 | 出典 |
|---|------|------------|---------|--------|------|
| A1 | FAISS `_id_map` と `ntotal` の不整合 | `recall` が `list index out of range` でクラッシュ | 内部 ID マッピングと実際の FAISS index の長さが乖離 | 境界チェック追加 + `compact(rebuild_faiss=True)` で再構築 | Operations-Troubleshooting.md T2 |
| A2 | FAISS 孤立ベクトル蓄積 | `hard_delete` / `archive` 後もベクトルが FAISS に残留、性能劣化 | FAISS はインプレース削除をサポートしない | `compact()` で物理削除 | Plans-Backend-Phase-A-B-C.md TD1 |
| A3 | アーカイブ済みノードが波動伝播にヒット | アーカイブ済みノードのベクトルが毎クエリの重力波伝播に捕まる、アーカイブ数に比例した線形コスト増 | FAISS 内のベクトルが残っている | `compact(rebuild_faiss=True)` | Plans-Backend-Phase-A-B-C.md TD2 |
| A4 | マルチプロセス間 FAISS index 乖離 | プロセス A の `remember` がプロセス B の FAISS index に即時反映されない | 各プロセスが独立した FAISS インスタンス + メモリキャッシュを持つ | 重要書き込み後に `compact` を定期実行 | Architecture-Concurrency.md C1 |
| A5 | FAISS ファイル上書き競合 | 2 プロセスが同時終了時に `.faiss` を同時保存、後勝ちでデータロス | プロセス間で FAISS 永続化の調停なし | `compact` で単一出力点に集約 | Architecture-Concurrency.md C2 |
| A6 | `.ids` ファイル破損 | ID マッピングファイルが FAISS index と同期不全 | 非アトミック書き込み、クラッシュ時の保存中書き込み、マルチプロセス競合 | `faiss_index.search` に防御的境界チェック + `compact` で完全再生成 | Architecture-Concurrency.md C3 |

**教訓**: FAISS は追加専用であり、削除・更新の整合性はアプリケーション側で管理する必要がある。`compact` を定期実行する運用が必須。

### カテゴリ B: SQLite 並行性の問題

| # | 事例 | 何が起きたか | 根本原因 | 解決策 | 出典 |
|---|------|------------|---------|--------|------|
| B1 | SQLite ロックエラー（`database is locked`） | 複数 MCP サーバープロセスからの同時書き込みでロックエラー | SQLite デフォルト `busy_timeout=0`（即時失敗） | `PRAGMA busy_timeout=30000`（30秒待機）を設定 | Operations-Troubleshooting.md T3 |
| B2 | TTL 有効期限切れノードがメモリに残留 | 長期稼働デーモンで期限切れ TTL ノードがキャッシュに残り続ける | `expire_due_nodes` が起動時のみ実行される設計だった | `compact` またはライトビハインドループに統合 | Plans-Backend-Phase-A-B-C.md TD3 |
| B3 | 異常終了によるダーティ状態ロスト | Ctrl+C / クラッシュで直近 5 秒の動的状態（mass, temperature, displacement）が消失 | ライトビハインドキャッシュのフラッシュ間隔が 5秒、未フラッシュ分はメモリのみ | グレースフルシャットダウンで全フラッシュ。動的状態は後続クエリで自然再構築 | Architecture-Concurrency.md C4 |

**教訓**: SQLite WAL + busy_timeout は並行書き込みを安全にするが、各プロセスのキャッシュは独立。キャッシュのリロードまで不整合を許容する設計が必要。

### カテゴリ C: 設計判断の誤りと是正

| # | 事例 | 何が起きたか | 根本原因 | 解決策 | 出典 |
|---|------|------------|---------|--------|------|
| C1 | 単一埋め込み空間の限界 | Phase 1 評価で、単一空間への重力変位が破壊的・不可逆的と判明 | 元の埋め込みを直接変更すると FAISS index 整合性が崩れる | **二重座標系** 導入：元の埋め込み（不変）+ 仮想座標（重力変位） | Architecture-Overview.md D1 |
| C2 | `graph_boost` のスコア加算が無意味 | グラフベースのスコアボーナスを追加しても有意なランク変化なし | 単純なスコア加算では検索結果の順序に影響を与えられなかった | `graph_boost` 廃止 → 重力変位統合（ノードを物理的に移動）に置換 | Architecture-Overview.md D2 |
| C3 | 変位の暴走 | 制限なしでは重力変位がノードを無関係な領域へ「脱出」させる | 高次元空間での非有界変位 | `max_displacement_norm=0.3` のハードキャップ + フックの法則による復元力 | Architecture-Overview.md D3 |

**教訓**: 物理アナロジーは強力だが、パラメータの境界条件（ハードキャップ、復元力）を設計段階で組み込まないと暴走する。

### カテゴリ D: オーバクレイム（過剰な主張）の是正

| # | 事例 | 何が起きたか | 根本原因 | 解決策 | 出典 |
|---|------|------------|---------|--------|------|
| D1 | 「数学的に同一」のオーバクレイム | README/SKILL.md が重力更新則を「Heavy ball SGD と数学的に同一」と記載 | 厳密な数学的同一性ではなく、**解釈的前提**（retrieval score を確率的勾配シグナルと見なす）の下での対応関係だった | 「項ごとの対応関係」「構造的に対応する」に修正。明示的な「前提となる解釈の約束事」セクション追加 | Git: f9bd645 (Phase A) |
| D2 | README の計測値と主張の混同 | TTT 対応を強く主張しつつ根拠となる計測値が見えない | Phase 2 評価数値が研究文書にあったが README に反映されていなかった | 「What we measured vs what we're claiming」セクション追加（計測値・主張・未解決問題を分離） | Git: c4b4de8 (Phase B) |

**教訓**: 学術的・技術的な主張は「計測」「解釈」「推測」を明確に分離しないと信頼を損なう。特に「同一（identical）」という言葉は慎重に使う。

## 4. 抽出されるメタパターン（再発防止の原則）

### 4.1 「動いているように見える」の罠
- FAISS のベクトルは追加後、表面上は正常に動くが、削除・アーカイブの整合性は別レイヤーの問題（A1-A6）
- 101 テスト通過 ≠ 出力フォーマット正しい（E1）
- **対策**: インデックスの健全性を `compact` で定期確認。E2E テスト層の独立。

### 4.2 「単純な変更」の多層的影響
- プロジェクト名変更 → install path, git, wiki, URL, 設定ファイル...（F1-F6）
- 新パラメータ追加 → 呼び出し元破壊（G2）
- 新カラム追加 → 既存 DB 起動不能（H1）
- **対策**: 変更影響のチェックリスト化。監査フェーズの必須化。

### 4.3 言葉の選択が信頼を決める
- 「同一（identical）」→ 対応関係に修正（D1-D3）
- **対策**: 主張は「計測」「解釈」「推測」に分離。強い言葉（identical, exactly, always）は使用前に三思。

### 4.4 物理アナロジーの両刃の剣
- 変位の暴走（C3）、gravity merge の暴走（T5）、basins of attraction のエージェント捕捉（O3）
- **対策**: 物理パラメータには常にハードキャップと復元力を設定。`max_displacement_norm` のような安全弁。

### 4.5 デフォルトの危険性
- `busy_timeout=0`（即時失敗）→ B1
- FAISS orphan vectors（自動削除なし）→ A2
- TTL 有効期限のサイレント発動 → T8
- **対策**: デフォルト値の前提を文書化。安全なデフォルト（fail-safe）の設計。

## 6. 結論

GaOTTT プロジェクトの失敗事例は **「動いているように見える中に潜む不整合」** と **「単純な変更の多層的影響」** という2つのメタパターンに集約される。前者は FAISS キャッシュ乖離・テストの盲点・TTL のサイレント発動として現れ、後者はリネーム罠・API 破壊・スキーマ移行の問題として現れた。

物理アナロジーの設計は GaOTTT の核心だが、変位の暴走や merge の暴走が示すように **安全弁（ハードキャップ + 復元力）なしに物理を信頼してはならない**。この原則はソフトウェア設計全般にも当てはまる：**美しい抽象化には必ず境界条件の防御を添える**。
