# SKILL.md 改良計画

> 対象: `SKILL.md`（リポジトリルート）および `.claude/skills/ger-rag/SKILL.md`
> 言語ポリシー: **本ドキュメント（保守計画）は日本語、SKILL.md 本体は英語**で記述する。
> 最終更新: 2026-04-21

## 0. なぜこの改良が必要か

GER-RAG は単なる KV ストアや検索インデックスではなく、**LLM（特に Claude）の外部長期記憶**として機能することを意図している。
現在の SKILL.md は API の使い方は説明できているが、**「どのような姿勢で・どのタイミングで・なぜ使うのか」**という運用思想が薄い。
LLM 視点からの自発的な発火条件が曖昧なため、結果的に保存し忘れたり、過剰保存になったりする。

本改良は、SKILL.md を「単なるツールのリファレンス」から「記憶運用のプロトコル仕様書」へ昇華させることを目的とする。

## 1. 現状の差分整理

| 項目 | `SKILL.md`（ルート） | `.claude/skills/ger-rag/SKILL.md` |
|---|---|---|
| `name` | ✅ あり | ✅ あり |
| `description` | ❌ **欠落** | ✅ あり |
| 本文 | ほぼ同一 | ほぼ同一 |

→ ルート側の `description` 欠落が最優先の修正項目。
→ 2ファイル維持か単一化（シンボリックリンク等）かは別途検討。当面は **両方を手で同期**する運用とし、改良完了後に一本化方針を決める。

## 2. フロントマター必須項目チェックリスト

Anthropic Skill 仕様（および本プロジェクトでの慣例）に基づく必須・推奨項目:

- [ ] `name` — スキル識別子（`ger-rag-memory`）
- [ ] `description` — **1行で「いつ・なぜ呼ぶか」を LLM が判断できる説明**
  - 悪い例: 「GER-RAG メモリツール」
  - 良い例: 「セッションをまたぐ長期記憶。会話冒頭の文脈復元、重要判断の保存、コンパクション直前の退避、過去の失敗との照合に使う」
- [ ] `version` — セマンティックバージョン（運用開始するなら）
- [ ] `tools`（任意）— このスキルが提供する MCP ツール名のリスト
- [ ] `triggers`（任意・将来）— 自動発火条件の宣言

## 3. 運用思想セクションの追加（新規）

SKILL.md 冒頭の "What this is" を拡張し、以下の比喩を明記する。

### 3.1 LLM のグリア細胞・アストロサイト的役割

- **ニューロン（Claude のトークン推論）** が前面で考える間、**アストロサイト（GER-RAG）** は裏で:
  - 過去の発火パターン（recall 履歴）を蓄積し、関連記憶を**重力で引き寄せて発火準備**しておく
  - 不要になった記憶を**剪定**する（dormant の検出と forget 提案）
  - ニューロン間の同期（過去の自分の判断と現在の自分の対話）を媒介する
- 単なる「外部 DB」ではなく、**思考を支持・栄養補給する組織**として位置付ける

→ この比喩を `## What this is` の直下に英語で 1 段落追加する。

## 4. 「いつ使うか」の網羅的プロトコル化

現行の "When to use" は API 対応表に留まっているので、**発火トリガー集**として書き直す。

### 4.1 必ず `recall` するタイミング

- 新しいセッションの**最初のユーザー発話を受けた直後**
- ユーザーが「前回」「以前」「この前」等の時系列指示語を含む発話をしたとき
- **エラーに遭遇したとき**（過去の同種トラブルシュートを引く）
- 重要な設計判断をしようとしているとき（過去の決定との一貫性確認）

### 4.2 必ず `remember` するタイミング

- ユーザーが好み・制約・禁止事項を明示した瞬間
- 試行錯誤の末に**問題が解決した瞬間**（成功体験は失敗と同等以上に価値がある）
- 失敗した・間違えた・撤回した判断（`source="agent"`, `tags=["mistake","retracted"]`）
- 反復思考で結論が変わったとき（旧結論と新結論を両方保存し、関係を `supersedes` で繋ぐ — ※本体改修依存）
- **コンパクション直前**: 会話が長くなってきた・要約圧縮の兆候を感じたら能動的に退避
- 「これは未来の自分宛のメモだ」と感じたとき（`source="agent"`, `tags=["letter-to-future-self"]`）

### 4.3 `explore` を使うべき瞬間

- 行き詰まったとき（同じアプローチでループしている自覚があるとき）
- 異分野・別プロジェクトの経験を借りたいとき
- ユーザーから「面白いアイディアない？」と聞かれたとき

### 4.4 `reflect` の儀式化

- セッション終了時に `reflect(aspect="hot_topics")` で当日の重力増加を眺める
- 週次で `reflect(aspect="dormant")` を行い、forget 候補をユーザーに提示する（※本体改修依存）

## 5. パターンセクションの拡張

### 5.1 既存パターン（維持・微調整）

- コンテキスト圧縮時の記憶退避
- 作業開始時の文脈復元
- 判断の記録
- トラブルシューティングの記録
- ユーザーの好みや制約の記録
- 創発的なアイデア探索

### 5.2 新規追加パターン

#### A. 過去の自分との対話モード（反復思考）

```
# 思い直しが発生した時のプロトコル
1. 現在検討中の判断と関連する過去判断を recall
2. 過去の Claude が下した結論を要約として提示
3. 「前回の自分はこう言ったが、今の状況では妥当か？」を内省
4. 結論が同じ → 強化（その記憶への暗黙の重力寄与）
5. 結論が変わった → 旧結論を retracted タグで残し、新結論を保存
   （※将来 supersedes リレーションが実装されたら明示的に繋ぐ）
```

#### B. Thinking ログの伏線退避

`<thinking>` ブロック内で「これは今は使わないが将来効きそう」と感じた仮説を、`source="hypothesis"` で短期保存する。
（※TTL 付き短期記憶は本体改修依存。実装前は `tags=["hypothesis","ephemeral"]` で代替運用）

```
remember(
  content="今回は採用しなかったが、recall に学習可能な温度パラメータを持たせる案。理由: ユーザーごとの探索選好に適応できそう",
  source="agent",  # 将来は source="hypothesis"
  tags=["hypothesis","ephemeral","explore-design"]
)
```

#### C. 自分への手紙

未来の自分が同じ状況で読み返すことを想定したメモ。

```
remember(
  content="次にPlotly 3Dの色制御で詰まったら、まず marker.line ではなく marker.color の RGBA alpha を疑え",
  source="agent",
  tags=["letter-to-future-self","plotly"]
)
```

#### D. forget 提案フロー（儀式）

```
1. reflect(aspect="dormant") を実行
2. 結果をユーザーに提示し「これらは剪定してよいか」を尋ねる
3. ユーザーの判断を尊重して削除/保持を決定
   （※削除 API は本体改修依存。実装前は tags=["archived"] でフラグ運用）
```

#### E. エモーショナル重み付け（暫定タグ運用）

scorer 本体に情動次元が入る前の暫定として、タグで近似する:

- `tags=["emotion:relief"]` — スッキリ解けた成功体験
- `tags=["emotion:frustration"]` — 悔しかった失敗
- `tags=["emotion:surprise"]` — 予想外のつながり発見

将来的には scorer に組み込む（本体改修計画の §5 参照）。

## 6. Notes セクションの拡張

- `recall` のたびに重力が蓄積される旨を、**「使えば使うほど賢くなる」と明示**
- 同一内容の重複は SHA-256 でスキップされる旨はそのまま
- 「このスキル自体に対する気づきも GER-RAG に保存してよい」と再帰性を促す 1 行を追加

## 8. 言語ポリシーの明文化

本リポジトリのドキュメント言語ルールを SKILL.md 改良と合わせて以下に固定する:

- **SKILL.md**（LLM 向けスキル定義）: **英語**
  - 理由: スキルは Claude/他 LLM が直接読むため、訓練分布的に英語のほうがプロンプト効率が良い
- **保守・計画ドキュメント**（本ファイル等）: **日本語**
  - 理由: ユーザー（開発者）が読むため
- **README.md / README_ja.md**: 既存の二言語運用を維持
